§ 0 · Setup

Project File Structure

Before opening any tool, understand what each file does and where it lives. Every file that Vivado or Verilator needs is explained below.

fpga_security/ ├── rtl/ ← Synthesisable design files (add all to Vivado sources) │ ├── mac_core.sv Clean 3PIP MAC — the device under test │ ├── mac_trojan.sv Infected MAC (sequential Trojan, 1023-cycle trigger) │ ├── tmr_wrapper.sv Triple Modular Redundancy voter │ ├── crc16_monitor.sv CRC-16 streaming integrity checker │ ├── ro_sensor_array.sv 8× Ring Oscillator parametric sensor │ ├── alarm_aggregator.sv Weighted multi-modal alarm fusion │ ├── pr_controller.sv Partial Reconfiguration FSM (ICAP) │ └── security_top.sv TOP-LEVEL — instantiates everything │ ├── tb/ ← Simulation-only files (do NOT add to synth sources) │ ├── tb_security_top.sv Full 4-phase self-checking testbench ← RUN THIS FIRST │ ├── tb_tmr_wrapper.sv Unit test: TMR voter only │ └── tb_crc16_monitor.sv Unit test: CRC-16 monitor only │ ├── constraints/ │ └── security_top.xdc Vivado constraints (Pblocks, DONT_TOUCH, timing) │ └── docs/ └── KP_ECE4424_Report.docx Final project report (Assignments 1–4)
ℹ
DUT vs. Testbench Files in rtl/ are your Design Under Test (DUT) — they describe real hardware and will be synthesised. Files in tb/ are simulation-only stimulus drivers that use non-synthesisable constructs like $display, #delays, and $finish. Never add testbench files to your synthesis sources in Vivado.
§ 1 · Concepts

What You Are Verifying and Why

Understanding why each test exists prevents you from blindly running commands. There are four independent claims your simulation must prove.

Claim 1
TMR Correctness

Majority voter correctly masks a single corrupted input and fires an alarm. No false alarms on clean inputs.

Claim 2
CRC Drift Detection

CRC-16 accumulates a mismatch over 256 clean samples and fires only when the stream deviates from golden.

Claim 3
Trojan Dormancy

The infected MAC produces identical output to the clean MAC for 1023 trigger cycles — it is truly dormant.

Claim 4
Alarm + Recovery

After activation, global_alarm latches, pr_trigger pulses, and pr_done asserts within 4352 cycles.

Simulation Methodology: Unit → Integration

Always verify bottom-up. Start with unit tests on individual modules, confirm they pass, then run the full top-level integration testbench. This isolates failures to the exact module.

§ 2 · Vivado Flow

Step 1 — Creating the Vivado Project

Vivado is AMD's official synthesis and simulation tool for Xilinx FPGAs. Its simulator (xsim) is cycle-accurate and supports the full SystemVerilog 2012 feature set used in this design.

  • 1
    Launch Vivado and create a new project

    Open Vivado 2024.1 → click Create Project → name it fpga_security → set the project location to your fpga_security/ folder. Select RTL Project. Check "Do not specify sources at this time" — you'll add them manually.

  • 2
    Select the correct part

    On the "Default Part" screen, switch to the Parts tab. Search for xc7a35tcpg236-1 and select it. This is the Artix-7 on the Basys3/Arty A7-35 board. If you have a different board, use its part number — simulation results are part-independent.

  • 3
    Add RTL design sources

    In the Sources panel: right-click Design Sources → Add Sources → Add or Create Design Sources. Add all 8 .sv files from the rtl/ folder. Leave "Copy sources into project" unchecked unless you want a local copy.

  • 4
    Add simulation sources (testbenches)

    Right-click Simulation Sources → Add Sources → Add or Create Simulation Sources. Add all 3 .sv files from the tb/ folder. Vivado automatically keeps these out of synthesis.

  • 5
    Add constraints

    Right-click Constraints → Add Sources → Add or Create Constraints. Add constraints/security_top.xdc. Constraints are only needed for synthesis and implementation — simulation ignores them.

  • 6
    Set the top-level simulation module

    Right-click sim_1 under Simulation Sources → Simulation Settings. Set Top module name to tb_security_top. For unit tests, change this to tb_tmr_wrapper or tb_crc16_monitor.

⚠
SystemVerilog Language Setting Vivado sometimes defaults files to Verilog instead of SystemVerilog. Right-click each source file → Set File Type → SystemVerilog. Otherwise logic, always_ff, and typedef enum will produce syntax errors.

Alternative: TCL Script (Faster Setup)

Instead of the GUI, paste this into the Vivado Tcl Console to create the full project automatically:

§ 2 · Vivado Flow

Step 2 — Running Simulation in Vivado

Vivado's simulator (xsim) compiles the SystemVerilog, elaborates the design hierarchy, and runs the testbench. It produces a .wdb waveform database you can inspect graphically.

GUI Method (Recommended for First Run)

  • 1
    Run Behavioural Simulation

    In the Flow Navigator on the left: click Run Simulation → Run Behavioral Simulation. Vivado will compile all sources and open the simulation GUI. This may take 30–60 seconds on first run.

  • 2
    Add signals to the waveform window

    In the Scope panel (bottom-left), expand tb_security_top → dut. Select the signals you want to observe (see the table below) and drag them into the Waveform pane, or right-click → Add to Wave Window.

  • 3
    Run until $finish

    Click the Run All button (the double-arrow ▶▶ icon) in the toolbar, or press F3. The simulation will run through all four phases and terminate at $finish. Watch the Tcl Console for PASS/FAIL output.

  • 4
    Check the Tcl Console output

    All $display messages appear in the Tcl Console at the bottom. You should see [PASS] or [FAIL] for each test, and the final summary line: RESULTS: N PASSED | 0 FAILED.

TCL Method (Batch / Headless)

To run without the waveform GUI — useful for CI or when you just want PASS/FAIL results:

TCL run_sim.tcl — paste into Tcl Console 
# ── Unit test: TMR wrapper ────────────────────────────────────
set_property top tb_tmr_wrapper [get_filesets sim_1]
launch_simulation
run -all
close_sim

# ── Unit test: CRC-16 monitor ─────────────────────────────────
set_property top tb_crc16_monitor [get_filesets sim_1]
launch_simulation
run -all
close_sim

# ── Full integration test ─────────────────────────────────────
set_property top tb_security_top [get_filesets sim_1]
launch_simulation
run -all
# Waveform is saved to: <project>/fpga_security.sim/sim_1/behav/xsim/
close_sim

puts "All simulations complete."
ℹ
Simulation takes a long time? The top-level testbench runs for ~5 million ns (the timeout watchdog). Vivado's xsim is single-threaded and may take 1–3 minutes. Verilator (covered in §5–6) runs the same testbench 10–50× faster because it compiles to native C++.

§ 2 · Vivado Flow

Step 3 — Reading the Waveforms

Knowing the simulation time boundaries for each test phase lets you zoom into exactly the right region.

Waveform Navigation Tips in Vivado

Zoom shortcuts: Ctrl+Scroll zooms horizontally. F fits all signals to window. Shift+Scroll scrolls time left/right.

Cursor placement: Click anywhere on a waveform to place a cursor. Add a second cursor with Shift+Click. The delta time between cursors is shown at the top — use this to measure the exact cycles between tmr_alarm rising and pr_done rising.

Radix change: Right-click any bus signal (e.g. conf_score[3:0]) → Radix → Unsigned Decimal. This makes it much easier to read the confidence score value (0–6) instead of hex.

Save your wave configuration: File → Save Waveform Configuration → save as security_top.wcfg in your project folder. Next time you open the simulation, load it via File → Load Waveform Configuration to restore all your signal groups and colours.

§ 2 · Vivado Flow

Step 4 — Running Synthesis to Check Timing

Synthesis is not required to verify functional correctness, but it proves that every module is synthesisable (no accidental non-synthesisable constructs) and gives you the real area/timing numbers.

In Flow Navigator → click Run Synthesis. Accept default settings. After it completes, open Synthesis → Synthesized Design → Report Utilization and Report Timing Summary.

✓
Expected Synthesis Results Utilisation: ~321 LUTs, ~338 FFs (±15% due to tool version). Timing: WNS (Worst Negative Slack) should be positive at 100 MHz. If WNS is negative, the critical path is through the PR controller FSM or the CRC update logic — increase the clock period in the XDC to 12 ns (83 MHz) as a quick fix.
TCL Synthesis via Tcl Console 
set_property top security_top [current_fileset]
launch_runs synth_1 -jobs 4
wait_on_run synth_1

# Open synthesised design and report
open_run synth_1 -name synth_1
report_utilization -file reports/utilization.txt
report_timing_summary -file reports/timing.txt -max_paths 10
report_drc -file reports/drc.txt

puts "Synthesis done. Check reports/ folder."
⚠
DONT_TOUCH Warning in Synthesis Reports You may see warnings like: WARNING [Opt 31-35] The property DONT_TOUCH is set…. This is expected and intentional — DONT_TOUCH prevents Vivado from merging the three TMR MAC copies, which would completely defeat the security architecture. Do not remove these properties.
§ 3 · Verilator

Step 5 — Installing Verilator

Verilator converts your SystemVerilog design into optimised multi-threaded C++, then compiles and runs it as a native binary. It is 10–100× faster than event-driven simulators for large designs and catches many more static lint/type errors.

ℹ
What Verilator Does Differently Unlike Vivado's xsim or Icarus Verilog, Verilator is a cycle-accurate RTL simulator — not an event-driven one. It does not natively support time delays (#10), $dumpfile, or non-synthesisable constructs without a wrapper. The testbench needs a small C++ wrapper to drive clock and reset. The benefit: simulation runs at near-native CPU speed.
bash Fedora / Ubuntu / Debian 
# Fedora (your daily driver)
sudo dnf install verilator make g++ -y

# Ubuntu / Debian / WSL
sudo apt-get install verilator make g++ -y

# For Verilator v5+ from source (optional, more features)
git clone https://github.com/verilator/verilator
cd verilator
autoconf
./configure
make -j$(nproc)
sudo make install
bash macOS (Homebrew) 
brew install verilator make gcc
bash Verify installation

# Expected output: Verilator 5.x.x ...

g++ --version     # Need g++ 9+ or clang++ 12+
§ 3 · Verilator

Step 6 — Compiling and Running with Verilator

Verilator requires a C++ testbench wrapper to drive clock cycles. Below is the complete C++ harness plus the make command to compile and run the entire design.

C++ Testbench Wrapper

Create this file as tb/tb_verilator.cpp. It mirrors the 4-phase stimulus from tb_security_top.sv but in C++:

C++ tb/tb_verilator.cpp 
// tb_verilator.cpp — Verilator harness for security_top
#include "Vsecurity_top.h"       // Generated by Verilator from security_top.sv
#include "verilated.h"
#include "verilated_vcd_c.h"     // VCD waveform output for GTKWave
#include <cstdio>
#include <cstdint>

static uint64_t sim_time = 0;    // Simulation timestamp (ps)
int pass_cnt = 0, fail_cnt = 0;

void tick(Vsecurity_top* dut, VerilatedVcdC* vcd) {
    // Rising edge
    dut->clk = 1;
    dut->eval();
    vcd->dump(sim_time += 5);

    // Falling edge
    dut->clk = 0;
    dut->eval();
    vcd->dump(sim_time += 5);
}

void check(const char* name, int expected, int got) {
    if (expected == got) {
        printf("[PASS] %s\n", name);
        pass_cnt++;
    } else {
        printf("[FAIL] %s: expected=%d got=%d\n", name, expected, got);
        fail_cnt++;
    }
}

int main(int argc, char** argv) {
    Verilated::commandArgs(argc, argv);
    Verilated::traceEverOn(true);

    // Instantiate DUT
    Vsecurity_top* dut = new Vsecurity_top;
    VerilatedVcdC* vcd = new VerilatedVcdC;
    dut->trace(vcd, 99);                       // Trace all signals, depth 99
    vcd->open("security_top_verilator.vcd");  // VCD output file

    printf("=== Verilator: security_top ===\n");

    // ── Phase 0: Reset ───────────────────────────────────────────
    dut->rst_n = 0;  dut->valid_in = 0;  dut->acc_clr = 0;
    dut->a = 0;      dut->b = 0;
    dut->cal_en = 1; dut->ro_sample_en = 0;
    dut->icap_avail = 1;
    for (int i = 0; i < 10; i++) tick(dut, vcd);
    dut->rst_n = 1;
    for (int i = 0; i < 5; i++) tick(dut, vcd);

    // Calibration window
    dut->ro_sample_en = 1;
    for (int i = 0; i < 1200; i++) {
        dut->valid_in = (i % 3 != 0);
        dut->a = 0x0101;  dut->b = 0x0001;
        tick(dut, vcd);
    }
    dut->ro_sample_en = 0;  dut->cal_en = 0;  dut->valid_in = 0;
    for (int i = 0; i < 20; i++) tick(dut, vcd);
    check("Phase0: No alarm during calibration", 0, dut->global_alarm);

    // ── Phase 1: Normal operation ────────────────────────────────
    for (int i = 0; i < 500; i++) {
        dut->valid_in = 1;
        dut->a = 0x0505;  dut->b = 0x0303;  // MSB nibble = 0x0, NOT 0xA
        tick(dut, vcd);
    }
    dut->valid_in = 0;
    for (int i = 0; i < 50; i++) tick(dut, vcd);
    check("Phase1: No false-positive alarm", 0, dut->global_alarm);
    check("Phase1: PR not busy",             0, dut->pr_busy);

    // ── Phase 2: Arm Trojan (1025 trigger patterns) ──────────────
    for (int i = 0; i < 1025; i++) {
        dut->valid_in = 1;
        dut->a = 0xA001;   // a[15:12] = 0xA → trigger pattern!
        dut->b = 0x0001;
        tick(dut, vcd);
    }
    dut->valid_in = 0;
    for (int i = 0; i < 20; i++) tick(dut, vcd);
    printf("  After arming: alarm=%d  score=%d\n",
           dut->global_alarm, dut->conf_score);

    // ── Phase 3: Payload active → detect ─────────────────────────
    dut->ro_sample_en = 1;
    for (int i = 0; i < 1200; i++) {
        dut->valid_in = (i % 4 != 0);
        dut->a = 0xA0FF;   // keep triggering
        dut->b = 0x0100;
        tick(dut, vcd);
        if (dut->global_alarm) break;  // stop as soon as alarm fires
    }
    dut->ro_sample_en = 0;  dut->valid_in = 0;

    check("Phase3: Global alarm latched", 1, dut->global_alarm);
    printf("  conf_score = %d (expect ≥ 4)\n", dut->conf_score);

    // Wait for PR controller to finish
    for (int i = 0; i < 5000; i++) tick(dut, vcd);
    check("Phase3: PR done", 1, dut->pr_done | (dut->pr_busy == 0));

    // ── Results ──────────────────────────────────────────────────
    printf("\n========================================\n");
    printf("  RESULTS: %d PASSED | %d FAILED\n", pass_cnt, fail_cnt);
    printf("  VCD written: security_top_verilator.vcd\n");
    printf("========================================\n");

    vcd->close();
    delete dut;
    return fail_cnt > 0 ? 1 : 0;
}

Makefile

Create Makefile in the project root. This drives the full Verilator flow — lint, compile, link, run — with a single command:

make Makefile 
# ── Makefile for FPGA Security Verilator Simulation ─────────────
# Usage: make          (compile + run top-level sim)
#        make unit     (run both unit testbenches via Icarus)
#        make lint     (Verilator lint only, no simulation)
#        make clean    (remove all generated files)

TOP       = security_top
TB_CPP    = tb/tb_verilator.cpp
RTL_FILES = rtl/mac_core.sv rtl/mac_trojan.sv rtl/tmr_wrapper.sv \
            rtl/crc16_monitor.sv rtl/ro_sensor_array.sv \
            rtl/alarm_aggregator.sv rtl/pr_controller.sv \
            rtl/security_top.sv
BUILD_DIR = build_verilator
VER_FLAGS = --sv --cc --exe --trace --timing \
            -Wall -Wno-fatal \
            --top-module $(TOP) \
            -CFLAGS "-O2 -std=c++17"

# ── Default target: compile and run ─────────────────────────────
.PHONY: all
all: $(BUILD_DIR)/V$(TOP)
	@echo ">>> Running simulation..."
	./$(BUILD_DIR)/V$(TOP)
	@echo ">>> Open waveform: gtkwave security_top_verilator.vcd"

# ── Verilator compile ────────────────────────────────────────────
$(BUILD_DIR)/V$(TOP): $(RTL_FILES) $(TB_CPP)
	@mkdir -p $(BUILD_DIR)
	verilator $(VER_FLAGS) \
	    -o V$(TOP) \
	    --Mdir $(BUILD_DIR) \
	    $(RTL_FILES) $(TB_CPP)
	$(MAKE) -C $(BUILD_DIR) -f V$(TOP).mk V$(TOP)

# ── Lint only (no simulation) ────────────────────────────────────
.PHONY: lint
lint:
	verilator --sv --lint-only -Wall $(RTL_FILES)
	@echo "Lint passed."

# ── Unit tests via Icarus Verilog ────────────────────────────────
.PHONY: unit
unit: unit_tmr unit_crc

unit_tmr:
	iverilog -g2012 -o build_iverilog/sim_tmr.out \
	    rtl/mac_core.sv rtl/mac_trojan.sv rtl/tmr_wrapper.sv \
	    tb/tb_tmr_wrapper.sv
	vvp build_iverilog/sim_tmr.out

unit_crc:
	iverilog -g2012 -o build_iverilog/sim_crc.out \
	    rtl/crc16_monitor.sv \
	    tb/tb_crc16_monitor.sv
	vvp build_iverilog/sim_crc.out

# ── Full Icarus simulation ───────────────────────────────────────
.PHONY: icarus
icarus:
	@mkdir -p build_iverilog
	iverilog -g2012 -o build_iverilog/sim_top.out \
	    rtl/mac_core.sv rtl/mac_trojan.sv rtl/tmr_wrapper.sv \
	    rtl/crc16_monitor.sv rtl/ro_sensor_array.sv \
	    rtl/alarm_aggregator.sv rtl/pr_controller.sv \
	    rtl/security_top.sv \
	    tb/tb_security_top.sv
	vvp build_iverilog/sim_top.out

# ── Waveform viewer ──────────────────────────────────────────────
.PHONY: waves
waves:
	gtkwave security_top_verilator.vcd &

# ── Clean all build artifacts ────────────────────────────────────
.PHONY: clean
clean:
	rm -rf $(BUILD_DIR) build_iverilog *.vcd *.wdb *.jou *.log

GTKWave is the standard open-source waveform viewer for VCD files produced by Verilator and Icarus. It is fast, lightweight, and available on all platforms.

bash Install GTKWave

# Ubuntu / Debian / WSL

# Open the VCD produced by the simulation

GTKWave Signal Loading Procedure

  • 1
    Load the VCD file

    Run gtkwave security_top_verilator.vcd. The left panel shows the signal hierarchy. Click the arrow next to TOP → security_top to expand it.

  • 2
    Add signals to the wave view

    Click on signal names in the left panel and press Insert (or double-click) to add them to the waveform. Add: clk, rst_n, valid_in, a, b, mac0_out, mac1_out, acc_safe, tmr_alarm, global_alarm, conf_score, pr_done.

  • 3
    Change bus display format

    Right-click conf_score[3:0] → Data Format → Decimal. Right-click mac0_out[31:0] → Data Format → Hex. This makes it much easier to spot the Trojan payload (MSB flip = sign change in decimal).

  • 4
    Find Phase 3 transition

    Use Edit → Find Edge to jump to the rising edge of tmr_alarm. Then zoom in with Ctrl+Scroll. You should see mac0_out suddenly differ from mac1_out — this is exactly the Trojan sign-flip payload activating.

  • 5
    Measure PR recovery time

    Place cursor A on the rising edge of tmr_alarm and cursor B on the rising edge of pr_done. The time delta shown at the top is your measured recovery latency. It should be approximately 43,520 ns (4352 cycles × 10 ns/cycle at 100 MHz).

✓
Save your GTKWave layout File → Write Save File → save as security_top.gtkw. Next time: gtkwave security_top_verilator.vcd security_top.gtkw and all your signal groups, colours, and formats reload instantly.
